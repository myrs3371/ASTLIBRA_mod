### BUG-01 · `app.js` — `startup()` 完全被注释，启动逻辑失效

**位置**：`web/js/app.js` `mounted()` 钩子

**问题代码**：
```js
mounted() {
    // window.addEventListener('pywebviewready', () => this.startup());
    // if (window.pywebview) this.startup();
},
```

**后果**：
- `gameDetected` 默认值为 `true`，即使未检测到游戏目录也不会弹出警告遮罩
- 首次运行不会自动触发文本提取流程
- 用户只能通过手动切换到「对话文本」页看到「暂无数据」才能触发提取

**修复方案**：
```js
mounted() {
    window.addEventListener('pywebviewready', () => this.startup());
    if (window.pywebview) this.startup();
},
```

### BUG-02 · `api.py update_text()` — CSV 读写双重格式错位

**位置**：`frontend/api.py` `update_text()` 方法

#### 问题一：读取时缺少 `header=0`

```python
# 当前（错误）
df = pd.read_csv(
    csv_path, encoding="utf-8",
    names=["id_текста", "Offset_start", "JPN", "US", "ZH_CN", "ZH_TW", "KO", "ES", ""]
)
```

根据 Pandas 文档：传入 `names` 但未指定 `header` 时，默认等同于 `header=None`，即第一行（原本的列标题行）被当作数据行读入。`Offset_start` 列全部错位，`df.loc[df["Offset_start"] == int(offset)]` **永远命中不到正确行**，所有文本修改静默失败。

#### 问题二：写回时丢掉列标题且改变引号格式

```python
# 当前（错误）
df.to_csv(csv_path, index=False, encoding="utf-8", quoting=1, header=False)
```

- `header=False`：CSV 写回后**不含列标题行**
- `quoting=1`（`csv.QUOTE_ALL`）：所有字段加引号，与 `_ALOC.py` 原始输出格式不一致
- 下次 `read_csv.py` 用 `header=0` 读取时，第一行数据行被误认为列名，**全表数据错乱**

**修复方案**：
```python
df = pd.read_csv(
    csv_path, encoding="utf-8-sig",
    names=["id_текста", "Offset_start", "JPN", "US", "ZH_CN", "ZH_TW", "KO", "ES", ""],
    header=0   # ← 补上，跳过标题行
)
df[...] = new_text
df.to_csv(csv_path, index=False, encoding="utf-8-sig", header=True)  # ← 保留标题行，去掉 quoting=1
```

### BUG-03 · `text_importer.py restore_original()` — 直接复制 `_dec` 文件当 DAT

**位置**：`backend/services/text_importer.py` `restore_original()` 末尾

**问题代码**：
```python
# 将原始模板文件复制到用户DAT文件夹
shutil.copy2(self.localize_dec, self.localize_dat)
```

**后果**：  
`LOCALIZE_.DAT_dec` 的文本段起始偏移为 `478776`（原始格式），而游戏运行时实际读取的 `LOCALIZE_.DAT` 由 `_ALOC.py -p` 打包，文本段偏移为 `2688`（打包格式）。直接将 `_dec` 文件复制为 DAT 文件，格式不一致，游戏可能读取错乱。

该函数已经通过 `-e` 重新生成了原始 CSV，应当在此基础上再 `-p` 打包，而非直接 copy 文件。

**修复方案**：
```python
# 删除直接 copy 的一行，改为打包
result = subprocess.run(
    ['python', '_ALOC.py', 'LOCALIZE_.DAT_dec', '_temp_restore.csv', '-p'],
    cwd=Config.CORE_DIR
)
dat_temp = os.path.join(Config.CORE_DIR, 'LOCALIZE_.DAT')
shutil.copy2(dat_temp, self.localize_dat)
os.remove(dat_temp)
```

### BUG-04 · `text_extractor.py` — 所有 `subprocess.run` 缺少窗口抑制参数

**位置**：`backend/services/text_extractor.py`，共 4 处

**问题**：打包为 exe（`console=False`）后，子进程会弹出黑色命令行窗口，且输出无法被捕获用于错误判断。

**受影响的调用**：

| 方法 | 调用 |
|------|------|
| `unpack_dat_dxa` | `subprocess.run([Config.ASTLIBRA_DEC, dat_file], cwd=...)` |
| `patch_exe` | `subprocess.run(['python', self.patch_exe_script, ...], cwd=...)` |
| `extract_texts` | `subprocess.run(['python', '_ALOC.py', ..., '-e'], cwd=...)` |
| `pack_to_dat` | `subprocess.run(['python', '_ALOC.py', ..., '-p'], cwd=...)` |

**修复方案**（每处统一补全）：
```python
result = subprocess.run(
    [...],
    cwd=...,
    capture_output=True,          # ← 补上
    timeout=60,                   # ← 补上
    creationflags=subprocess.CREATE_NO_WINDOW  # ← 补上
)
```

### BUG-05 · `text_extractor.py patch_exe()` — 未检查 EXE 是否已修补

**位置**：`backend/services/text_extractor.py` `patch_exe()` 方法

**问题**：每次提取流程都无条件执行 `patch_exe.py`。第二次运行时，EXE 已修补，`patch_exe.py` 找不到目标字节序列，返回 `False`，导致提取第三步直接报错失败。

对比 `mod_manager.py activate_mods()` 里已有正确的检查逻辑：
```python
patched_bytes = bytes([0x90, 0x90, 0x90, 0x8B, 0x41, 0xFC, ...])
with open(exe_path, 'rb') as f:
    exe_data = f.read()
if patched_bytes not in exe_data:
    patch_exe(exe_path)
else:
    print("游戏EXE已修补，跳过")
```

**修复方案**：在 `text_extractor.py` 的 `patch_exe()` 方法中加入同样的已修补检查，复用 `core/patch_exe.py` 的 `patch_exe()` 函数而非通过子进程调用，避免重复代码。

### BUG-06 · `mod.js restoreAll()` — 提示文案与实际行为不符

**位置**：`web/js/pages/mod.js` `restoreAll()` 方法

**问题代码**：
```js
'确定要还原所有原版文件吗？\n\n这将删除以下文件夹：\n...\n\n然后重新解压 DAT_BACK.dxa 备份文件。'
```

**实际行为**：`restore_all()` 只将 `*_BACK.dxa` **重命名**为 `*.dxa`，并不重新解压任何文件。

**修复方案**：
```js

```

### BUG-08 · CSV 编码不一致（`utf_8_sig` vs `utf-8`）

| 文件 | 操作 | 编码 |
|------|------|------|
| `core/_ALOC.py` | 初始写入 CSV | `utf_8_sig`（含 BOM） |
| `core/read_csv.py` | 读取 CSV | `utf-8`（无 BOM） |
| `frontend/api.py update_text` | 读取 & 写入 CSV | `utf-8`（无 BOM） |

读取 BOM 文件时若用 `utf-8` 编码，BOM 字符（`\ufeff`）会残留在第一列列名中，可能引起列名匹配失败。建议统一为 `utf-8-sig`。
